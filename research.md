# Исследование: Telegram Mini Apps для сервисных каталогов

**Дата:** 22 мая 2026  
**Контекст:** Анализ конкурентов и паттернов перед разработкой Mini App для каталога услуг тарологa Ana Krista Goyya.

---

## 5 примеров Telegram Mini Apps в нише сервисных каталогов

---

### 1. DIKIDI

**Что делает:** SaaS-платформа для бьюти-бизнеса (салоны, частные мастера) с Telegram Mini App для записи. Мастера получают личный виджет-ссылку — клиент открывает её и попадает в Mini App внутри Telegram.

**Функциональность:** Выбор мастера → выбор услуги → календарь с временными слотами → подтверждение → уведомления-напоминания → онлайн-оплата.

**Сильные стороны:**
- Zero friction авторизация — Telegram передаёт данные автоматически, клиент ничего не вводит вручную
- Напоминания и подтверждения приходят прямо в Telegram-чат
- Поддержка команды мастеров, не только соло
- Широкая известность в СНГ — клиенты уже знакомы с системой

**Слабые стороны:**
- Полностью generic UI — каждый салон выглядит одинаково, ноль бренда
- Холодная утилитарная цветовая схема (синий/белый) — не подходит для эмоциональных ниш
- Нет портфолио или фото работ внутри флоу записи
- Устаревший UX выбора времени — плоский список слотов, а не визуальный календарь
- Нет апселла и допродаж
- Медленная загрузка из-за тяжёлого SaaS-бэкенда

**Что мы можем сделать лучше:**
- Персональный бренд с тёмной мистической эстетикой вместо generic-шаблона
- Показывать «что именно ты получишь» — описание, что включено в сессию, как проходит расклад
- Быстрая загрузка (Vite + минимальный bundle)

### 2. YClients (Telegram-интеграция)

**Что делает:** Доминирующий SaaS для записи в бьюти и фитнес (20 000+ бизнесов в России). С 2023 года — Telegram Mini App: кнопка «Записаться» в канале салона открывает Mini App прямо в Telegram.

**Функциональность:** Дерево услуг → список мастеров с фото → выбор даты → подтверждение → баланс бонусов → история визитов → отмена/перенос.

**Сильные стороны:**
- Профили мастеров детальнее, чем у DIKIDI — биография, фото
- Модуль лояльности — клиент видит свои баллы, это удерживает
- Отмена записи внутри приложения — нет лишних сообщений в поддержку
- Показывает длительность услуги — клиент знает, что записывается на 90 минут

**Слабые стороны:**
- UI ощущается как legacy-сайт, втиснутый в Telegram — кнопки не используют Telegram-стиль, нет адаптации к теме
- Не поддерживает dark mode — белый фон у dark-mode пользователей, визуальный разрыв
- Нет фото услуг — «Французский маникюр — 1500 ₽» без иллюстрации
- Нет индикатора прогресса в многошаговом флоу — клиент не знает, сколько шагов осталось
- Нет gift booking и share-флоу

**Что мы можем сделать лучше:**
- Dark mode с первой секунды — наша тёмная тема идеально ложится в Telegram
- Индикатор шагов («2 из 3») в процессе оформления
- Фото/иллюстрация для каждой услуги

### 3. Standalone-бот частного мастера (constructor-based)

**Что делает:** Частные мастера (нейлы, ресницы, брови, фотографы) создают собственные боты через Botmother, SendPulse или на заказ. Пример: nail-артист с 50K подписчиков делает `@krisnails_bot`, ссылка из Instagram-био открывает Mini App с портфолио и записью.

**Типичный функционал:** Фотогалерея работ → прайс → форма записи → свободные слоты → оплата депозита.

**Сильные стороны:**
- Личный бренд — UI использует цвета и эстетику мастера, не шаблон
- Портфолио на первом экране — фото работ конвертируют лучше любого текста
- Депозит встроен в флоу — уменьшает no-show в разы
- Тёплый personal tone: «Я» вместо «мы»
- Лёгкий и быстрый — нет SaaS-бэкенда

**Слабые стороны:**
- Нет синхронизации слотов в реальном времени — мастер обновляет вручную, возможны двойные записи
- Нет напоминаний — клиенты забывают, мастер теряет деньги
- Нет повторной записи — после визита не приходит «запишись снова через 3 недели»
- Нет истории клиента у мастера — примитивная или нулевая CRM
- Контент устаревает — бот показывает неактуальное расписание, если мастер уехал

**Что мы можем сделать лучше:**
- Автоматические напоминания через Telegram
- CTA «повторная запись» после завершённой сессии
- Синхронизация доступности через простую backend-логику

### 4. Barbershop / бьюти-сеть Mini App (TOPGUN-style)

**Что делает:** Сетевые барбершопы и салоны, у которых есть собственный Mini App для записи к конкретному мастеру. Полноценная система с inline-календарём, профилями мастеров и Telegram Payments.

**Функциональность:** Выбор мастера → услуга → слот в календаре → подтверждение прямо в чат-трид.

**Сильные стороны:**
- Inline-календарь быстрее телефонного звонка
- Профили мастеров снижают тревогу («а кто меня стрижёт?»)
- Подтверждение в тред — не теряется в email-спаме
- Повторная запись бесфрикционна — данные сохранены

**Слабые стороны:**
- Отмена записи зарыта или отсутствует — приходится звонить
- Нет нотификации «мастер задерживается»
- Описания услуг — стены текста без визуальной иерархии
- Сетка календаря на 375px неудобна — попасть в слот сложно
- Нет отзывов и чаевых внутри флоу

**Что мы можем сделать лучше:**
- После сессии авто-сообщение «Как всё прошло?» с оценкой в 1 тап
- День как горизонтальная полоска с прокруткой — не плотная таблица

---

### 5. Фотограф / индивидуальная студия

**Что делает:** Частный фотограф или маленькая студия — клиент смотрит портфолио, выбирает тип съёмки, слот, заполняет бриф, вносит аванс.

**Функциональность:** Портфолио-грид → тип сессии (портрет / семья / продукт) + длительность + цена + пример → слот → бриф-форма → оплата аванса.

**Сильные стороны:**
- Портфолио внутри WebView — самая сильная фича: клиент видит качество до оплаты
- Карточки типов съёмки с примером фото задают ожидания
- Бриф-форма сокращает DM-переписку («сколько человек», «стиль», «локация»)

**Слабые стороны:**
- Фото не оптимизированы — долгая загрузка убивает доверие сразу
- Нет стайл-квиза или мудборда — клиент не может выразить желаемую эстетику
- Перенос записи не поддерживается — нужно писать в DM
- Политика возврата не видна до оплаты — создаёт тревогу перед нажатием «Оплатить»
- Нет флоу доставки результата — переходит в ручную пересылку файлов

**Что мы можем сделать лучше:**
- Для таро: 3-вопросный «квиз намерения» перед каталогом — персонализирует опыт
- Политику возврата/условия — показывать inline рядом с кнопкой оплаты, не в отдельной странице
- Кнопка «Предложить другую дату» вместо тупика «нет слотов»

## Стандартные экраны Telegram Mini App для сервисного каталога

### Обязательные экраны

| Экран | Описание |
|---|---|
| **Home / Каталог** | Точка входа. Аватар и имя мастера, сетка/список услуг. `expand()` на загрузке, нет BackButton. |
| **Service Detail** | Главный конверсионный экран. Название, описание, цена (всегда видна), что включено, CTA через MainButton. |
| **Booking Form** | Максимум 2–3 шага. Имя авто-заполняется из `initData`. Вопрос / дата / формат. MainButton = «Далее» / «Отправить». |
| **Confirmation** | Thank you экран. Что происходит дальше («Анна напишет в течение 24 часов»). Haptic success. MainButton = «Вернуться в каталог». |

### Дополнительные экраны (если каталог растёт)

| Экран | Когда нужен |
|---|---|
| **Категории** | При 8+ услугах — chips/tabs над каталогом |
| **Корзина** | Если клиент может купить несколько товаров за раз |
| **История заказов** | Для повторных клиентов — «мои покупки» |
| **Профиль** | Telegram-имя из `initData`, бонусы (если есть) |

## Обязательные элементы Telegram Mini Apps SDK

### Критические вызовы при старте
```javascript
Telegram.WebApp.expand()          // Развернуть на весь экран — СРАЗУ
Telegram.WebApp.ready()           // Сигнал Telegram что загрузка завершена
Telegram.WebApp.setHeaderColor('#080614')     // Цвет статус-бара = наша тёмная тема
Telegram.WebApp.setBackgroundColor('#080614') // Фон Telegram = наша тёмная тема
```

### MainButton
- Единственная основная CTA на каждом экране
- `MainButton.setText('Записаться')` / `MainButton.show()` / `MainButton.hide()`
- `MainButton.showProgress()` во время async-операций — без этого клиенты тапают повторно
- Не дублировать HTML-кнопкой с той же функцией

### BackButton
- Показывать на всех экранах кроме Home: `BackButton.show()`
- Синхронизировать с навигацией в useState: `BackButton.onClick(handleBack)`
- Никогда не полагаться на browser back gesture в WebView

### HapticFeedback
- Лёгкий impact — тап на карточку услуги
- Medium impact — добавление в корзину
- Success notification — успешное бронирование/оплата
- Error notification — ошибки

### Telegram CSS переменные (тема)
```css
--tg-theme-bg-color
--tg-theme-text-color
--tg-theme-hint-color
--tg-theme-button-color
--tg-theme-secondary-bg-color
```
Для нашей фиксированной тёмной темы: инициализировать через SDK, чтобы не было белых вспышек.

### Данные пользователя (auth без формы)
```javascript
Telegram.WebApp.initDataUnsafe.user
// { id, first_name, username, language_code }
```
Авто-подставлять имя в формы — ключевое преимущество перед сайтом.

### Клавиатура и безопасные зоны
- Слушать `viewportChanged` — пересчитывать layout при открытии клавиатуры
- Использовать `viewportStableHeight` для форм с text input

### Диалоги
- **Запрещено:** `window.alert()`, `window.confirm()` — выглядят как браузерные попапы
- **Правильно:** `Telegram.WebApp.showPopup()`, `showConfirm()`, `showAlert()`

### Закрытие с подтверждением
```javascript
// В начале многошагового флоу:
Telegram.WebApp.enableClosingConfirmation()
// На экране подтверждения:
Telegram.WebApp.disableClosingConfirmation()
```

### Внешние ссылки
```javascript
// Всегда так — не через <a href>:
Telegram.WebApp.openLink(url)
Telegram.WebApp.openTelegramLink(url) // для t.me ссылок
```

## Паттерны нативного ощущения vs «сайт в Telegram»

| Нативный Mini App | Сайт в Telegram |
|---|---|
| `expand()` сразу, полный экран | Открывается маленькой карточкой |
| BackButton синхронизирован с навигацией | Собственная стрелка в своём header |
| Haptic feedback в ключевых моментах | Никакой обратной связи |
| MainButton для основного действия | CTA спрятан в скролле страницы |
| `showPopup()` для подтверждений | `window.alert()` — выбивает из контекста |
| Skeleton-загрузка | Белый экран или спиннер посередине |
| Тёмная тема с первого пикселя | Белая вспышка при открытии |
| Имя пользователя из `initData` | Форма «введите ваше имя» |
| Подтверждение закрытия в многошаговом флоу | Приложение молча закрывается на середине |
| Переходы через bottom sheet / slide | Перезагрузка страницы между экранами |

## Типичные ошибки, убивающие конверсию

1. **Нет явного действия на первом экране** — пользователь не понимает, что делать
2. **Требование контактов до показа ценности** — доверие ещё не установлено
3. **Скрытая цена** — «узнать стоимость» в сервисном каталоге = уход
4. **Нет контекста перед оплатой** — кнопка «Оплатить» без условий создаёт тревогу
5. **Тупик после покупки** — «Спасибо» без «что дальше»
6. **Неоптимизирован под 375px** — минимум 48px высота кнопок
7. **Нет MainButton.showProgress()** — клиент тапает повторно, создаёт дубли
8. **Generic копирайтинг** — «Submit» / «Confirm» вместо голоса бренда
9. **Нет follow-up** — после закрытия Mini App только одно сообщение в чате, которое теряется
10. **Не вызван `expand()`** — пользователь должен вручную разворачивать

## Белое пятно рынка

Все существующие решения жертвуют **визуальным повествованием** ради механики записи. Платформы с сильным портфолио (standalone-боты) конвертируют лучше, но не имеют инфраструктуры. Платформы с сильной инфраструктурой (DIKIDI, YClients) выглядят как банковские приложения.

**Незанятая позиция:** Mini App который:
- Загружается мгновенно
- Уважает dark mode с первой секунды
- Открывается с эмоционального visual-контента (описание, изображение услуги)
- Доводит до записи/покупки за 3 тапа
- После покупки говорит «что дальше» в голосе бренда

Именно это мы строим.

## Концепция v3+: Мультимастерная платформа

*Идея зафиксирована как направление развития. Не реализуется в текущей версии.*

### Суть идеи

Один Telegram-бот (или платформа) обслуживает **нескольких мастеров**. У каждого мастера своя аудитория клиентов. Клиенты разных мастеров не пересекаются и не видят друг друга.

### Как выглядит интерфейс

В верхней части бота — переключатель ролей:

```
┌─────────────────────────────────┐
│  [ Я мастер ]  [ Я клиент ]     │  ← два таба вверху
└─────────────────────────────────┘
```

**Вид клиента** — каталог услуг выбранного мастера, запись, оплата, история заказов.

**Вид мастера** — входящие заявки, расписание, статистика продаж, настройки профиля и услуг.

### Схема данных

```
Платформа
  ├── Мастер А (Ana Krista)
  │     ├── Услуги А
  │     └── Клиенты А (видят только услуги А)
  └── Мастер Б (другой таролог)
        ├── Услуги Б
        └── Клиенты Б (видят только услуги Б)
```

### Что нужно для реализации

| Компонент | Зачем |
|---|---|
| База данных | Хранить мастеров, клиентов, заявки |
| Авторизация | Определять кто мастер, кто клиент |
| Панель мастера | Заявки, расписание, статистика |
| Мультитенантность | Изоляция данных между мастерами |
| Платёжная система | Автоматическая оплата и выплаты мастерам |

### Когда имеет смысл строить

Когда Ana Krista или другие мастера захотят:
- Объединиться в одном боте (школа, агентство)
- Дать другим мастерам использовать ту же платформу
- Масштабироваться и получать % от продаж других мастеров

### Текущий статус

Текущая версия — **соло-платформа для одного мастера** (Ana Krista). Это правильный первый шаг: проверить спрос, отработать UX, собрать клиентскую базу. Мультимастерная платформа — следующий этап после подтверждения модели.

## Экспертная оценка

*Три точки зрения на Mini App каталог услуг тарологa Ana Krista Goyya.*

---

### 👁 UX-дизайнер

#### Ошибки, которые делают новички

**1. Начинают с каталога, а не с эмоции.**
В таро пользователь приходит не «купить услугу» — он приходит с тревогой, вопросом, желанием понять ситуацию. Первый экран должен встречать его как человека, а не как покупателя. Типичная ошибка — сразу карточки с ценами. Нужен короткий «якорь»: кто Ana, почему ей можно доверять, и только потом — выбор.

**2. Слишком много вариантов сразу.**
Если на первом экране 12 карточек услуг — это паралич выбора. Пользователь не знает, с чего начать, и уходит. Решение: структура с 3–4 категориями («Консультации», «Курсы», «Материалы», «Подписка») и по 2–3 услуги в каждой.

**3. Нет объяснения, что именно происходит после оплаты.**
В эзотерике покупатель боится: «А вдруг деньги уйдут, и ничего не будет?». Стандартное «Заказ принят» не работает. Нужно: что конкретно получит клиент, когда и в каком формате («Я напишу тебе в этот чат в течение 24 часов и пришлю файл»).

**4. Игнорируют пустые состояния.**
Нет свободных слотов — что видит пользователь? Пустой экран, белое поле, ошибка? Каждый dead end — потеря клиента. Нужен friendly empty state: «Сейчас записей нет — напиши мне напрямую» с кнопкой.

**5. Форма с лишними полями.**
Telegram уже знает имя пользователя. Запрашивать его снова — сигнал, что приложение «не знает» пользователя. Максимум нового: один текстовый вопрос («Что тебя беспокоит?»). Всё остальное — из `initData`.

#### Что обязательно должно быть

| Элемент | Почему |
|---|---|
| Фото и имя Аны на главном экране | Доверие продаётся через личность, не через услугу |
| Цена на каждой карточке | Скрытая цена = подозрение. Всегда показывать. |
| «Что входит» в детали услуги | В нише таро клиент не знает, чего ждать — нужно объяснить |
| Skeleton-лоадер, не спиннер | Ощущение скорости, пока грузятся данные |
| Confirmation screen с «что дальше» | Снять тревогу после оплаты |
| Haptic success при подтверждении | Эмоциональная точка на завершении действия |
| BackButton синхронизирован с навигацией | Иначе пользователь «застревает» |

#### Что бесит пользователей

- Белая вспышка при открытии — приложение «вываливается» из тёмного Telegram
- Нужно вручную разворачивать Mini App (не вызван `expand()`)
- Кнопка «Купить» ниже fold — приходится скроллить, чтобы найти действие
- После оплаты — тишина. Прошло 10 секунд, ничего не произошло, непонятно, что делать
- Приложение закрылось случайным свайпом в середине заполнения формы
- Текст «Оформить заказ» / «Подтвердить» — нет голоса бренда, ощущение интернет-магазина

### 📣 Маркетолог

#### Что заставит записаться

**1. Конкретный результат, а не описание услуги.**
Плохо: «Расклад на отношения».
Хорошо: «После расклада ты поймёшь, куда движется эта связь и стоит ли её продолжать».
В эзотерике покупают не процесс, а ответ на вопрос.

**2. Ана как личность, не как сервис.**
Короткий абзац от первого лица: «Я Ана, тарологMY. Работаю с Таро Уэйта, за 5 лет провела 2000+ консультаций» — конвертирует в разы лучше, чем безликий каталог.

**3. Social proof прямо в карточке.**
«⭐ 4.9 · 340 отзывов» под названием услуги — убирает сомнения ещё до перехода в детали. В v1 можно сделать статикой из реальных отзывов Telegram-канала.

**4. Entry point с низким барьером.**
Первым в каталоге должен быть самый дешёвый продукт (PDF, мини-расклад за 390–490 ₽). Клиент, купивший что-то недорогое, с вероятностью 3–5× купит дорогую консультацию.

**5. Scarcity без манипуляции.**
«В ближайшие 2 недели осталось 3 слота» — если это правда, работает хорошо. Не «осталось 1 место!» каждый день — это убивает доверие надолго.

**6. Привязка к моменту.**
«Расклад в период Новолуния», «Работаем с энергией Весов» — контекстуальные акценты на главном экране повышают ощущение своевременности и актуальности.

#### Элементы, повышающие конверсию

| Элемент | Где размещать | Эффект |
|---|---|---|
| Badge «Хит» / «Популярное» | На карточке в каталоге | Социальное доказательство, снижает паралич выбора |
| «Что ты получишь» (список) | В детали услуги | Снимает неопределённость, повышает ценность |
| Цена + формат одной строкой | В карточке каталога | «Онлайн · 60 мин · 3 500 ₽» — нет вопросов |
| Отзыв прямо под кнопкой «Записаться» | В детали услуги | Последний push перед тапом |
| Post-purchase CTA | На Confirmation screen | «Хочешь разобрать расклад глубже? →» |
| Кнопка «Поделиться» | В каталоге и деталях | Органический рост через Telegram forward |
| Персональное приветствие | На Home экране | «Привет, Катя!» из `initData` — тепло и нативность |

#### Воронка, которая работает

```
Канал Аны → кнопка «Открыть приложение»
  → Home: фото Аны + короткий призыв + каталог
    → Карточка услуги: результат + что входит + отзыв + цена
      → Форма: 1 вопрос + имя уже заполнено
        → Оплата
          → Confirmation: что дальше + post-purchase CTA
            → Бот присылает подтверждение в чат
```

Каждый переход — одно решение. Никаких разветвлений, никаких «а может, сначала посмотрю другое».

### ⚙️ Telegram-разработчик

#### Ограничения, о которых нужно знать заранее

**1. Нет нативного date/time picker.**
В Mini App нет компонента выбора даты уровня iOS/Android. Придётся строить свой на React. Горизонтальный скроллируемый список дней — проще и надёжнее, чем полноценный calendar grid (который сложно сделать удобным на 375px).

**2. iOS WebView и Android WebView ведут себя по-разному.**
На iOS клавиатура «поднимает» весь viewport — нужно слушать `viewportChanged` и перестраивать layout. На Android этого может не быть. Тестировать на обоих.

**3. Нет push-нотификаций из Mini App.**
Уведомления клиенту (подтверждение, напоминание) можно отправлять только через Telegram-бота. Mini App сам по себе ничего не отправляет после закрытия. Это означает: нужен бот рядом с Mini App.

**4. localStorage очищается.**
Нельзя надёжно хранить данные между сессиями. Для истории заказов нужен backend. В v1 — ок, не хранить ничего, всё подтверждение идёт через бота в чат.

**5. Telegram Payments и Stars — разные флоу.**
Telegram Stars (внутренняя валюта) — самый простой путь оплаты, не нужен платёжный провайдер. Обычные карты (RUB) — нужен BotFather invoice + подключённый провайдер (Tinkoff, Stripe). Для старта Stars проще, но аудитория должна иметь Stars на счету.

**6. Swipe-to-close конфликтует с горизонтальными жестами.**
Если делать горизонтальный свайп в multi-step форме (e.g., «проведи для следующего шага»), он будет конфликтовать с жестом закрытия Mini App. Лучше использовать вертикальные переходы или кнопки.

**7. Внешние шрифты — CORS и скорость.**
Google Fonts загружаются медленно или блокируются. Лучше использовать system fonts или inline base64 для одного кастомного шрифта. У нас Tailwind — можно ограничиться `font-sans`.

#### Что работает хорошо — использовать активно

- **`initDataUnsafe.user`** — авторизация без форм, это killer feature Mini Apps
- **MainButton** — всегда виден, Telegram его оптимизировал для iOS/Android; надёжнее любой HTML-кнопки
- **HapticFeedback** — дешёвый способ сделать приложение «живым»; 3 события стратегически = ощущение нативного приложения
- **`showPopup()` / `showConfirm()`** — нативные диалоги, не рвут контекст
- **`openTelegramLink()`** — правильный переход в канал Аны, в профиль бота, в чат
- **Тёмная тема** — большинство power users Telegram используют dark mode; наш `#080614` — это конкурентное преимущество, не просто эстетика

#### Что лучше не делать

| Избегать | Почему | Альтернатива |
|---|---|---|
| React Router / history API | Конфликт с BackButton Telegram | useState + стек экранов вручную (уже используется) |
| `window.alert()` / `confirm()` | Выбивает из контекста, выглядит как браузер | `Telegram.WebApp.showPopup()` |
| `<a href>` для внешних ссылок | Открывает внутри WebView, ловушка | `openLink()` / `openTelegramLink()` |
| Анимации с `transform: translateX` на корневом элементе | Конфликт со swipe-to-close | Анимировать дочерние элементы |
| Bundle > 300KB gzipped | Медленная загрузка в WebView | Code splitting, lazy imports |
| Горизонтальный scroll-snap между экранами | Конфликт с закрытием | Вертикальные переходы, bottom sheets |
| Хранить sensitive данные в localStorage | Очищается, небезопасно | Всё через бота / подтверждение в чат |
| Свой header с кнопкой «Назад» | Двойная навигация, путаница | Только Telegram BackButton |

#### Архитектурное решение для v1 (без backend)

Для MVP можно обойтись без сервера совсем:

```
Mini App (React + Vite)
  → Пользователь выбирает услугу
  → Telegram.WebApp.sendData(JSON) отправляет данные боту
  → Бот получает данные, присылает счёт или подтверждение клиенту
  → Ане приходит уведомление о новой заявке
```

Это работает без базы данных. Ана получает заявки в Telegram как сообщения. Минус: нет истории заказов для клиента. Плюс: ноль инфраструктуры в v1.
